# 项目规则

## 适用范围

- 当前工作区使用项目本地 skill，目录为 `[.codex/skills](./.codex/skills)`。
- 优先使用项目本地 skill，不使用同名的用户级或全局 skill。
- 如果项目流程发生了会影响重复性工作的变化，需要在同一轮修改里同时更新 `AGENTS.md` 和对应的本地 skill，避免两边规则漂移。

## 工作模式

当前项目至少存在两种不同的工作模式，后续处理任务时必须先判断属于哪一种，再应用对应规则。

### 模式 A：资料整理模式

适用于：

- 更新主文档
- 整理住院文档
- OCR 报告
- 报告重命名
- 生成 PDF
- 维护检查报告目录

这一模式下，优先遵循本文件中“主文档、住院文档、OCR、PDF、skill 同步”等规则。

### 模式 B：系统开发模式

适用于：

- 开发查询管理系统
- 编写前后端代码
- 设计数据库与索引层
- 构建搜索、时间线、图表、指标页
- 调整前端 UI 组件和交互

这一模式下，除了通用代码规范外，还应遵循下面的“系统开发规则”。

## 主文档

- 面向医生的主文档是 `[柯灵基本信息.md](./柯灵用/柯灵基本信息.md)`。
- 主文档中保留当前准确的用药剂量。
- 当前用药方案变化时，既要更新主文档里的当前剂量，也要保证前面的调药时间轴仍然能解释为什么会调整到现在的方案。
- 更新“目前治疗方案”时，尽量保留明确的“截至日期”或“按某次住院时整理”的说明。
- 对慢性病后续长期参考价值较高的监测指标，可以保留在主文档中。
- 只对某一次住院有意义、后续参考价值不高的短期化验结果，不要大量堆进主文档。

## 住院文档

- 每次住院或当前状态相关的时效性内容，放到对应的住院/检查报告文件夹里单独整理成 markdown。
- 每份住院整理应包含：事件经过、主要症状、即时处理、分组化验结果解读、原始报告索引。
- 通过 OCR 得到但不能完全确认的数值，要明确写成“约”，并保留原图链接。

## OCR 与报告文件

- 先 OCR，再给 `微信图片_*.jpg`、`微信图片_*.png` 这类泛文件名改名。
- 资料整理模式下，OCR 统一通过本地 `medical-report-organizer` 里的 `Invoke-MedicalOcr.ps1` 调用 `app/backend/.venv` 内的 PaddleOCR，不再默认回退到 Windows OCR。
- 只有在日期和报告类型基本确认后，才重命名报告文件。
- 报告文件名优先采用 `YYYY-MM-DD_报告名称.ext` 这种格式。
- 如果日期或数值不确定，不要假装是精确值。

## Skill 同步

- 本地医疗整理 skill 是 `[medical-report-organizer](./.codex/skills/medical-report-organizer)`。
- 本地主文档 PDF 导出 skill 是 `[main-doc-pdf-export](./.codex/skills/main-doc-pdf-export)`。
- 如果文档结构、命名规则、整理逻辑、解释习惯发生变化，要同步更新对应本地 skill。

## PDF 规则

- 修改主 markdown 文档后，不代表必须立刻重建 PDF。
- 修改 `[柯灵基本信息.md](./柯灵用/柯灵基本信息.md)` 后，需要询问用户是否现在刷新对应 PDF。
- 只有在用户明确确认后，才生成或覆盖主文档 PDF。
- 主文档 PDF 默认输出到 `[柯灵基本信息.pdf](./柯灵用/柯灵基本信息.pdf)`，除非用户另外指定路径。

## 系统开发规则

- 查询管理系统是“查询优先”，不是“表单录入优先”。
- 后续资料录入仍然以“用户与 AI 对话，AI 修改文档”的方式为主，系统代码不要强行改造成复杂手工录入后台。
- 系统的数据源优先是项目内现有 markdown、图片、PDF 和整理稿，代码要围绕这些真实文件做索引和查询。
- 查询系统启动约定：前端默认可直接在 `app/frontend` 下运行 `pnpm run dev`；后端统一通过 `scripts/Start-Backend.ps1` 启动，确保使用 `app/backend/.venv`，且该虚拟环境统一固定为 Python 3.12；同时启动前后端时优先使用 `scripts/Start-QueryApp.ps1`。
- 查询系统部署约定：开发阶段继续优先保留后端 `/api/*` 接口模式；对外分享阶段优先采用“静态分享模式”，在构建时自动导出 JSON 数据和原始资料静态文件，避免为了临时分享额外依赖常驻云端 Python 服务。
- 前端 UI 优先使用 **Ant Design**。
- 前端 markdown 展示优先使用 **Ant Design X** 的 `@ant-design/x-markdown`。
- 涉及 markdown 页面渲染时，默认直接采用 `@ant-design/x-markdown`，不把通用 markdown 解析库作为首选默认方案，除非后续明确出现 Ant Design X 无法满足的能力缺口。
- 前端整体展示生态优先采用 Ant Design 相关方案，不优先混入风格差异大的其他 UI 体系。
- 图表和可视化也优先采用 Ant Design 生态方案，默认先考虑 `@ant-design/plots`，只有在明显能力不足时再评估其他图表库。
- 系统开发模式下，前端默认技术组合优先按 `antd` + `@ant-design/x-markdown` + `@ant-design/plots` 这一组生态协同推进，避免页面层混入过多割裂的展示方案。
- 编写前端页面时，优先考虑信息查询、筛选、浏览、回溯，不追求花哨交互。
- 在适合的地方加入图表辅助说明，例如指标趋势图、时间线图、简单统计图，但图表不能替代表格和文本说明。
- 所有关键查询页面都要考虑移动端响应式兼容，至少保证手机上可以完成查看总览、搜索、查看时间线、查看住院详情、查看当前用药这些核心操作。
- 如果系统开发过程中形成了新的页面结构、目录规范、组件约束、查询范式，也需要同步更新相关本地 skill 或方案文档。
