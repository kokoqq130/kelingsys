# 查询管理系统技术方案

## 1. 方案目标

基于当前项目现有资料结构，设计一套本地优先、查询优先的技术实现方案。  
系统需要在不改变现有“通过 AI 对话维护文档”的工作方式前提下，提供稳定的查询与浏览能力。

技术方案的前提是：

- 原始资料继续保存在项目文件夹中
- markdown、图片、PDF 仍然是事实来源
- 系统通过解析这些文件建立索引
- 系统本身优先服务查询，而非录入

## 2. 总体架构建议

建议采用三层结构：

1. **文件层**
   当前工作区中的 markdown、图片、PDF、报告目录
2. **索引层**
   负责扫描文件、解析内容、抽取结构化数据、建立搜索索引
3. **查询展示层**
   提供本地 Web 页面，用于搜索、时间线查看、指标查看、文档回溯

推荐形态：

- 后端：Python
- 本地数据库：SQLite
- 前端：React + Vite + Ant Design + Ant Design X

## 2.1 前端 UI 约束

前端 UI 方案优先使用 **Ant Design**。

原因：

1. 适合快速搭建信息密度较高的后台/查询型界面
2. 表格、表单、时间线、抽屉、标签、卡片、树形结构等组件成熟
3. 对当前这种“查询优先、录入较少”的系统非常契合
4. 后续做筛选、详情抽屉、统计面板、时间线页面时，开发效率高

实现建议：

- 基础 UI 组件优先使用 `antd`
- 页面布局优先使用 `Layout`、`Menu`、`Card`、`Tabs`、`Table`、`Descriptions`、`Timeline`、`Drawer`
- 搜索与筛选优先使用 `Form`、`Input`、`Select`、`DatePicker`
- 状态标识优先使用 `Tag`、`Badge`、`Alert`
- 如无明确能力缺口，不默认引入风格差异较大的其他 UI 体系承担主界面展示

文档参考：

- Ant Design 官方聚合文档：`https://ant.design/llms-full.txt`

这份文档适合在后续编码时快速查询组件能力、API 和可用模式。

## 3. 为什么推荐 Python

当前项目已经天然更偏向 Python 生态：

1. 项目里已经存在 Python 脚本能力
2. OCR、文档解析、结构化抽取都更适合 Python
3. 后续如果要引入 NLP / 本地 embedding / 检索，也更方便
4. 和当前本地脚本型工作流衔接自然

因此建议：

- 用 Python 负责扫描、解析、索引、API
- 用前端负责界面展示

## 4. 数据源设计

### 4.1 原始数据源

当前系统的数据源不是数据库，而是工作区文件：

- `柯灵用/柯灵基本信息.md`
- `柯灵用/柯灵基本信息.pdf`
- `柯灵用/检查报告/**/*.jpg|png|pdf`
- `柯灵用/检查报告/**/住院整理.md`
- 其他后续整理文档

### 4.2 数据源原则

- 文件是第一事实来源
- 数据库只保存“索引结果”和“结构化缓存”
- 当文件变更时，索引层重新同步数据库

这样可以避免两套数据源长期漂移。

## 5. 模块划分

### 5.1 文件扫描模块

职责：

- 扫描项目根目录
- 识别 markdown、图片、PDF
- 识别主文档、住院文档、检查报告、技能脚本
- 生成文件清单

建议输出字段：

- file_id
- relative_path
- file_type
- ext
- last_modified_at
- size
- hash

### 5.2 文档解析模块

职责：

- 解析主文档 markdown
- 解析住院整理 markdown
- 提取标题、段落、时间点、链接、图片
- 提取事件和指标片段

MVP 阶段建议基于规则解析，不先做重模型抽取。

需要区分两层：

1. **后端结构化解析**
   - 用于抽取时间、事件、药物、指标、文件关联
   - 这一层仍以 Python 规则解析为主
2. **前端 markdown 展示**
   - 用于在界面中高质量渲染 markdown 内容
   - 这一层优先使用 Ant Design X 提供的 `@ant-design/x-markdown`

也就是说：

- “结构化抽取” 与 “页面渲染” 是两件事
- 后端继续负责抽取结构化数据
- 前端展示 markdown 时，优先使用 Ant Design X 生态

### 5.2.1 Markdown 展示库选择

前端 markdown 展示默认方案：

- `@ant-design/x-markdown`

原因：

1. 属于 Ant Design X 生态，和整体前端方案一致
2. 适合 markdown 场景下的流式渲染与扩展
3. 支持代码高亮、图表插件、扩展组件等能力
4. 适合后续做“主文档预览”“住院文档预览”“搜索命中片段展示”

参考文档：

- `https://ant-design-x.antgroup.com/x-markdowns/introduce-cn`

建议用法：

- 主文档详情页：使用 `XMarkdown` 渲染 markdown 正文
- 住院整理页：使用 `XMarkdown` 渲染正文
- 搜索结果详情：可渲染局部 markdown 或跳转到全文预览
- 若后续出现兼容性或扩展性问题，再单独评估补充其他 markdown 工具，但不作为首版默认选型

### 5.3 结构化抽取模块

职责：

- 从文档中抽取：
  - 时间
  - 事件类型
  - 药物
  - 剂量
  - 指标名称
  - 数值
  - 单位
  - 是否住院
  - 是否为 OCR 近似值

建议先采用“规则 + 正则 + 少量模板”的方式，不要一开始就重度依赖 LLM 在线抽取。

### 5.4 搜索索引模块

职责：

- 对文档全文建立全文检索
- 对事件、指标、药物建立结构化索引

建议使用：

- SQLite FTS5 做全文检索

优势：

- 本地部署简单
- 无需额外服务
- 足够支撑当前项目规模

### 5.5 查询 API 模块

职责：

- 提供前端查询接口
- 支持筛选、排序、分页、详情查看

建议用：

- FastAPI

典型接口：

- `/api/overview`
- `/api/timeline`
- `/api/events`
- `/api/medications`
- `/api/labs`
- `/api/search`
- `/api/files`

### 5.6 前端展示模块

职责：

- 展示概览
- 时间线
- 检查指标页
- 文档详情页
- 搜索结果页

建议重点做“能快速查”，而不是“页面很复杂”。

建议页面映射如下：

- 总览页：`Card + Descriptions + Tag + Alert`
- 时间线页：`Timeline + Card + Drawer`
- 指标页：`Table + Statistic + Segmented + Tabs`
- 文件资料页：`Table + Tree + Drawer`
- 搜索结果页：`Input.Search + List/Table + Highlight`

## 7. 文档到结构化数据的映射策略

### 7.1 主文档

主文档主要抽取：

- 患者基础信息
- 当前用药
- 主要诊断
- 发作时间线
- 长期指标摘要

### 7.2 住院文档

住院文档主要抽取：

- 住院日期
- 住院原因
- 本次事件经过
- 本次检查结果分组
- 本次初步解读

### 7.3 文件命名

命名本身也可作为结构化线索：

- `2026-02-04_吡仑帕奈血药浓度监测.jpg`
  可直接抽出：
  - 日期
  - 检查名称

所以系统应优先利用规范文件名。

## 8. 查询能力设计

### 8.1 全文搜索

支持搜索：

- 呕吐
- 地西泮
- 吡仑帕奈
- 同型半胱氨酸
- 住院

返回：

- 命中的文档段落
- 命中文件
- 对应时间
- 跳转入口

### 8.2 结构化搜索

支持：

- 查所有发作事件
- 查所有住院
- 查所有调药记录
- 查某个药物的变化历史
- 查某个指标历史结果

### 8.3 时间线视图

按时间汇总：

- 发作
- 住院
- 调药
- 检查

### 8.4 指标视图

针对重点长期指标做专页：

- 同型半胱氨酸
- 吡仑帕奈血药浓度
- 血氨
- 乳酸

## 8.5 数据可视化

虽然系统定位是“查询优先”，但仍然建议加入适量图表，帮助更快理解趋势，而不是只给文本和表格。

建议加入的图表场景：

1. **指标趋势图**
   - 同型半胱氨酸历史变化
   - 吡仑帕奈血药浓度变化
   - 血氨 / 乳酸等关键指标变化
2. **事件时间轴可视化**
   - 发作、住院、调药、检查事件按时间线展示
3. **当前状态摘要图**
   - 当前药物种类数
   - 近一年住院次数
   - 近一年发作次数

图表原则：

- 图表是辅助解释，不要喧宾夺主
- 图表必须能回到原始数据与原始文档
- 所有关键图表都要保留文本/表格替代视图，避免只靠图形表达

技术建议：

- 可先采用 `@ant-design/plots`
- 复杂图表场景再评估 `echarts-for-react`
- MVP 阶段先做折线图、时间线图、简单柱状图
- 先解决“趋势可见”，不要一开始追求复杂交互

## 9. 更新机制设计

### 9.1 更新方式

后续数据更新仍然通过 AI 改文档完成。  
系统不负责“录入”，只负责“感知文件变化并重建索引”。

### 9.2 更新触发方式

MVP 阶段建议：

- 手动点击“重建索引”
- 或启动系统时自动扫描变更文件

### 9.3 变更检测

可通过以下方式判断文件是否变化：

- 相对路径
- 最后修改时间
- 文件 hash

## 10. 推荐实现阶段

### 第一阶段：查询 MVP

目标：

- 能搜
- 能看时间线
- 能看指标列表
- 能跳原始文件

包含：

- 文件扫描
- markdown 解析
- SQLite 建库
- FTS 搜索
- 基础前端页面

### 第二阶段：结构化增强

目标：

- 查询更准
- 指标更清晰
- 事件关联更完整

包含：

- 更好的规则抽取
- 指标趋势视图
- 药物时间线视图
- 事件与文件关联增强

### 第三阶段：智能检索增强

目标：

- 更自然地问问题
- 能返回更聚合的答案

可考虑：

- 本地 embedding
- RAG 检索
- 结合本地 LLM 做查询增强

但这一阶段不是当前必须项。

## 11. 技术选型建议

### 后端

- Python 3.14
- FastAPI
- SQLite / SQLite FTS5
- Pydantic

### 前端

- React
- Vite
- Ant Design
- Ant Design X
- TanStack Router 或 React Router
- TanStack Query
- `@ant-design/plots` 或 ECharts（用于趋势图与简单统计图）

前端生态优先级建议：

1. **基础组件与页面框架**：`antd`
2. **markdown 展示**：`@ant-design/x-markdown`
3. **图表展示**：优先 `@ant-design/plots`，复杂场景再考虑 ECharts

整体原则是：

- 界面展示优先使用 Ant Design 相关生态
- markdown 渲染默认使用 Ant Design X 提供的方案
- 尽量减少混用过多风格不同的第三方 UI 方案

## 11.1 响应式与移动端兼容

虽然这个系统以查询为主，但移动端兼容必须纳入设计范围。

原因：

1. 你后续很可能会在手机上临时查询当前方案或某次住院内容
2. 医生沟通场景中，移动端快速打开资料很常见
3. 如果只做桌面端，很多高频查询场景会被限制

建议原则：

- **桌面优先设计，但必须移动端可用**
- 页面在手机端至少应支持：
  - 查看总览
  - 搜索
  - 查看时间线
  - 查看某次住院详情
  - 查看当前用药

实现建议：

- 使用 Ant Design 的栅格系统与响应式断点
- 桌面端以多栏布局为主，移动端自动折叠为单栏
- 表格在移动端要提供降级方案：
  - 横向滚动
  - 或切换为卡片列表视图
- 复杂筛选器在移动端优先放进 `Drawer`
- 长文详情页移动端要保证字号、留白、点击区域合适

MVP 要求：

- 支持常见手机宽度下的浏览
- 不要求首版就做成“移动端极致体验”
- 但不能出现主要页面不可用、文字拥挤、表格完全看不清的问题

### 文档解析

优先：

- 自研规则解析
- Python 标准库

后续可补：

- markdown 解析库
- 更强的文本抽取工具

## 13. 风险与应对

### 13.1 OCR 不稳定

风险：

- 某些报告 OCR 不准确

应对：

- 把 OCR 结果视为候选信息
- 保留原图跳转
- 在结构化数据里加 `is_approximate`

### 13.2 文档格式未来变化

风险：

- 主文档和住院文档格式未来会继续演进

应对：

- 解析器尽量基于“标题语义 + 文件命名 + 规则”实现
- 不依赖过度脆弱的固定行号

### 13.3 早期结构化不足

风险：

- 部分内容暂时只能全文搜索，不能很好结构化

应对：

- 接受 MVP 阶段“全文搜索优先”
- 后续逐步增强规则抽取

## 14. 当前推荐结论

技术上最推荐的路线是：

1. **继续保留现有文件夹和 AI 整理流程**
2. **新增一套 Python + SQLite 的索引层**
3. **再加一个本地 Web 查询界面**
4. **先做查询，不做复杂录入**

这是和你当前工作方式最兼容、也最容易逐步落地的方案。
