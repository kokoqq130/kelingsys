# 查询管理系统数据库表结构草案

## 1. 文档目标

本文档用于定义查询管理系统 MVP 阶段推荐的数据库表结构。

设计目标：

- 支持文档驱动
- 支持全文搜索
- 支持时间线
- 支持指标查询
- 支持原始文件回溯

推荐数据库：

- SQLite

## 2. 设计原则

1. 文件是事实来源，数据库是索引缓存
2. 尽量允许“部分结构化”，不要要求所有内容一次性抽取完整
3. 优先满足查询，不追求高度范式化

## 3. 核心表

### 3.1 files

用于存储工作区内文件的基本元数据。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| relative_path | TEXT UNIQUE | 相对路径 |
| file_name | TEXT | 文件名 |
| ext | TEXT | 扩展名 |
| file_type | TEXT | markdown / image / pdf / other |
| size_bytes | INTEGER | 文件大小 |
| sha1 | TEXT | 文件内容摘要 |
| updated_at | TEXT | 文件修改时间 |
| created_at | TEXT | 入库时间 |

### 3.2 documents

用于存储可解析文档的正文和类型。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| file_id | INTEGER | 对应 files.id |
| doc_kind | TEXT | main_summary / admission_note / report_index / other |
| title | TEXT | 文档标题 |
| content_text | TEXT | 文档纯文本内容 |
| summary_text | TEXT | 可选的系统摘要 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 3.3 events

用于存储时间线事件。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| event_date | TEXT | 事件日期 |
| event_time_text | TEXT | 时间文本 |
| event_type | TEXT | seizure / admission / medication_adjustment / lab / other |
| title | TEXT | 事件标题 |
| summary | TEXT | 事件摘要 |
| detail_text | TEXT | 事件详情 |
| is_hospitalized | INTEGER | 是否住院 |
| source_document_id | INTEGER | 来源文档 |
| source_file_id | INTEGER | 来源文件 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 3.4 medications

用于存储药物与剂量信息。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| name | TEXT | 药物名称 |
| category | TEXT | 抗癫痫 / 代谢病 / 补充用药 |
| dose_text | TEXT | 剂量原文 |
| route | TEXT | 给药方式 |
| frequency | TEXT | 频次 |
| start_date | TEXT | 开始日期 |
| end_date | TEXT | 结束日期 |
| is_current | INTEGER | 当前是否在用 |
| note | TEXT | 备注 |
| source_document_id | INTEGER | 来源文档 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 3.5 lab_results

用于存储重点检查结果。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| result_date | TEXT | 检查日期 |
| test_group | TEXT | 检查分组 |
| test_name | TEXT | 检查名称 |
| result_text | TEXT | 结果原文 |
| numeric_value | REAL | 数值结果 |
| unit | TEXT | 单位 |
| reference_text | TEXT | 参考范围 |
| status | TEXT | high / low / normal / unknown |
| is_approximate | INTEGER | 是否 OCR 近似值 |
| source_document_id | INTEGER | 来源文档 |
| source_file_id | INTEGER | 来源文件 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 3.6 document_file_links

用于存储文档和文件之间的关联，比如住院文档关联多张报告图。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | INTEGER PK | 主键 |
| document_id | INTEGER | documents.id |
| file_id | INTEGER | files.id |
| relation_type | TEXT | image_ref / source_report / related_pdf / other |
| created_at | TEXT | 创建时间 |

## 4. 搜索相关表

### 4.1 documents_fts

使用 SQLite FTS5 建立全文检索表。

建议字段：

- title
- content_text
- summary_text

### 4.2 events_fts

用于事件全文搜索。

建议字段：

- title
- summary
- detail_text

## 5. 索引建议

建议创建以下索引：

- `idx_files_relative_path`
- `idx_events_event_date`
- `idx_events_event_type`
- `idx_medications_name`
- `idx_medications_is_current`
- `idx_lab_results_result_date`
- `idx_lab_results_test_name`
- `idx_lab_results_test_group`

## 6. 表之间的关系

### 主关系

- `files` 1 -> 0/1 `documents`
- `documents` 1 -> N `events`
- `documents` 1 -> N `medications`
- `documents` 1 -> N `lab_results`
- `documents` N <-> N `files` 通过 `document_file_links`

## 8. 示例数据映射

### 示例 1：主文档中的当前用药

来源：

- `柯灵基本信息.md`

可抽取到：

- medications 表中的多条当前药物记录

### 示例 2：2026-03-22 住院整理

来源：

- `2026-03-22住院整理.md`

可抽取到：

- events：一次 admission
- events：一次 seizure
- lab_results：血气、乳酸、电解质、血氨、血糖、肝肾功等

### 示例 3：报告图片

来源：

- `2026-03-22_血氨测定.png`

可入库：

- files 一条记录
- 通过 `document_file_links` 关联到对应住院文档

## 9. 同步策略建议

### 文件变更后如何处理

建议流程：

1. 扫描文件
2. 比较 hash 或修改时间
3. 找到变化文件
4. 重新解析变化文件
5. 更新对应 documents / events / medications / lab_results

### 删除文件时如何处理

建议：

- files 中标记删除或直接删记录
- 同步删除或重建关联表

## 10. 当前建议结论

MVP 阶段的数据库不需要追求过度复杂，重点是：

- `files`
- `documents`
- `events`
- `medications`
- `lab_results`
- `document_file_links`

只要这几张表稳定下来，就足以支撑第一版查询系统。
